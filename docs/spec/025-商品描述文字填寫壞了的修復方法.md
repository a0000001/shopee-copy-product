> ## 一、 問題脈絡與演進歷程

> ### #shopee-desc-history-timeline
> #### 開發修復時間軸
1. **SPA 點擊翻頁修復 (cde6666)**：完成賣家列表 SPA 分頁抓取能力。
2. **文字填寫超時 (6b31040c)**：發現在 `fillAll()` 中包了 `uploadMediaAsync()`，下載與注入 9 張圖片與影片需 20~40 秒，且 `fillAndSaveSingle` 又發了一次獨立的 `uploadMedia`，導致媒體重複處理。媒體吃光 45 秒文字填寫時間配額，產生超時、分頁關閉。
3. **描述與數量填寫修復**：修復文字階段剝離媒體後，進一步修復富文字編輯器 (`[contenteditable="true"]`) 與數量 (stock/quantity) 欄位之 DOM 選擇器與 `setNativeValue` 賦值邏輯。

> ## 二、 第一階段：文字填寫超時與雙重媒體修復

> ### #shopee-desc-media-decoupling
> #### 媒體與文字解耦機制

> > **根因**：`fillAll()` 內部調用 `uploadMediaAsync()`，導致媒體下載（下載 9 張圖片 + 1 部影片）與上傳被重複執行兩次。`fillAll()` 耗時吃掉 45 秒文字配額超時。

> > **修正細節**：
> > 1. `extension/batch-upload.js`：發送 `fillProductData` 訊息時傳入 `skipMedia: true`，讓 `fillAll()` 專注於分類選取與文字欄位填寫（<10 秒完成）。
> > 2. `extension/content.js`：`fillProductData` handler 呼叫 `fillAll().then(sendResponse)`，同時寫入 `window._sgcFillState` 提供雙向驗證。
> > 3. 媒體改由第二階段 `uploadMedia` 獨立處理，兩者權責分離。

> ## 三、 第二階段：商品描述富文字與數量欄位修復

> ### #shopee-desc-stock-selector-fix
> #### 富文字編輯器與數量欄位增強

> > **根因**：
> > 1. 蝦皮賣家中心商品描述欄位為富文字編輯器（`[contenteditable="true"]` 或 `.rich-text-editor`），原程式僅判斷 `.ql-editor`，導致在非 Quill 編輯器上記錄 `setNativeValue` 失敗或無效。
> > 2. `findFieldByLabel` 與 `fillAll` 對「商品數量」與「商品描述」欄位僅使用了限制性較強的類名 selector (`input.eds-input__input` / `.ql-editor`)，當蝦皮前端元件改版或使用 EDS Input Number (`input`) 時導致無法獲得 DOM 元素。

> > **修正細節** (`extension/content.js`)：
> > 1. **`setNativeValue()` 增強**：
> >    - 擴充判斷條件：`input.classList.contains('ql-editor') || input.getAttribute('contenteditable') === 'true' || input.isContentEditable || input.classList.contains('rich-text-editor')`
> >    - 統一觸發 `input`, `change`, `blur` 事件（帶 `bubbles: true, cancelable: true, composed: true`）。
> >    - 針對標準 `input` 增加 `try-catch` 降級備援：`try { proto.set.call(input, value) } catch (e) { input.value = value }`
> > 2. **`findFieldByLabel()` 增強**：
> >    - `fieldIdMap` 增加 `'數量': 'stock'` 與 `'庫存': 'stock'` 別名。
> >    - `fieldId` 比對成功後，查詢備選 selector 納入 `input`, `textarea`, `[contenteditable="true"]`, `.ql-editor`。
> > 3. **`fillAll()` 多重 selector 降級備援**：
> >    - **商品描述**：`findFieldByLabel('商品描述')` ➔ `[data-product-edit-field-unique-id="description"] .ql-editor` ➔ `[data-product-edit-field-unique-id="description"] [contenteditable="true"]` ➔ `[data-product-edit-field-unique-id="description"] .rich-text-editor` ➔ `textarea`
> >    - **商品數量**：`findFieldByLabel('商品數量')` ➔ `findFieldByLabel('數量')` ➔ `findFieldByLabel('庫存')` ➔ `[data-product-edit-field-unique-id="stock"] input.eds-input__input` ➔ `[data-product-edit-field-unique-id="stock"] input` ➔ `input[placeholder*="數量"]` ➔ `input[placeholder*="庫存"]`

> ## 五、 2026-07-23 實測後發現：真正的 Root Cause

> ### #shopee-real-root-cause-jul23

#### 推論錯誤：不是蝦皮改 DOM

2026-07-23 用 CDP（Chrome DevTools Protocol）連接真實賣家中心（類別：電腦與周邊配件 > 軟體），**逐項測試所有 selector**，結果：

**99% 的 selector 仍然命中**，唯一變更只有 `installment` → `productInstallmentStatus`（且 content.js 用 `*=` 匹配，依然相容）。

詳見 001 文件 §10.3 重驗證結果：
- `file:///S:/projects/shopee-copy-product/docs/data/mcp%20devtools%20蒐集的蝦皮資料/001-賣家中心新增商品（seller-new-product-dom-analysis）.md`

#### 真正的 Root Cause（兩個低級錯誤）

##### 1. Bracket Imbalance — content.js 語法錯誤（主因）

`content.js` 有一段 dangling `async function fillCategoryAsync` 殘碼（原第 948~955 行），僅有 function 宣告開頭、沒有完整主體，導致整支 content script **語法錯誤無法執行**。所有舊版 checkout 回來也都帶著這個殘碼，所以「全部失敗」。

**為什麼 6865629 當時是正常的？**
推測當時 Chrome 載入的是舊版 content.js（無殘碼的快取版本），或該殘碼是在某個後續 commit 才被引入但未被發現。

##### 2. Popup.js 不等待填入完成 — UI 誤報 0/0

`popup.js:269` 發送 `fillProductData` 後，content.js 立刻回 `{ ok: true, status: 'started' }`（填入才剛開始），popup 就檢查 `resp.results`：

```js
const ok = (resp.results || []).filter(r => r.ok).length  // → 0
const total = (resp.results || []).length                  // → 0
```

永遠顯示「✅ 完成 0/0 個欄位」。使用者看到 0/0 以為完全沒作用，但其實 fillAll 正在背景執行。

**修正**：改為 polling `checkFillStatus`（每 500ms × 120 次 = 60 秒），等 `status === 'done'` 再顯示真實結果。
