## 我建議的改寫原則與規劃
不改核心事實、不改嚴謹語氣、不改你已經驗證過的資訊。只動 **結構** 和 **節奏**。
### 每個商品的統一結構（草案）
```
[第一段：痛點鉤子] → 1-2句，讓讀者覺得「這在說我」
[第二段：核心功能] → 3-5個bullet point，只寫用戶最關心的
[第三段：誰適合] → 強化身份認同
[第四段：信任/售後] → 整合硬體檢測、售後描述、版本更新等
```
### 具體改什麼、不改什麼
| 改 | 不改 |

| hashtag移除 | 價格 |

## AI 處理關卡（可開關）

### 動機

開發者寫完 ps_product_description 之後，多一道可選的 AI 處理關卡。啟動時它會對原文做一些結構性調整；不啟動時完全不影響，原文原樣送出。

### 開關設計

- 預設：**關閉**
- 啟動方式：在上架腳本或 JSON 裡加一個 flag，例如 `"ai_rewrite": true`
- 每個商品可以獨立開關（而非全有全無）

### AI 做的項目（可逐項開關）

| 項目 | 說明 | 建議預設 |
|---|---|---|
| 移除內嵌hashtag | 去掉 `#AI #NSFW #ComfyUI` 這類 tag，避免跟我們自己掛的重複 | ON |
| 移除硬體規格段落 | 去掉描述中「最低顯存：8GB」「建議N卡」這類硬體資訊，因為這些已在 computer_specs 結構化欄位裡，重複寫在描述裡佔篇幅又難維護 | ON |
| 過量 emoji 瘦身 | 連續 emoji（🔥✅🎥）超過 3 個的，減到 1-2 個 | ON |
| 雙標點符號修正 | 「！！」「？？」還原成單一標點 | ON |
| 移除購買須知模板 | 刪除「購買須知」「想知道你的電腦能不能跑這款AI模型？」「使用AI硬體相容性檢測工具」等開場白，因為 computer_specs 已是獨立欄位 | ON |
| 過度修辭降溫 | 「最強」「終極」「統治級」同一商品出現 ≥2 次時，保留 1 次，其餘換中性詞 | OFF（需人工確認） |
| 結構格式化 | 強制用統一的 section 順序（痛點→功能→對象→售後） | OFF（變動較大） |

### 實作建議

描述改寫用 LLM prompt 做，不寫死 regex 邏輯：

```
你是一個電商商品描述編輯器。你的任務是根據以下規則修改商品描述，不做任何事實改動。

規則：
1. 移除所有 #hashtag 格式的文字
2. 移除所有硬體規格描述（RAM、VRAM、CPU、顯卡型號等）
3. 連續 emoji 超過 3 個的，保留前 1 個
4. 將「！！」「？？」改為「！」「？」
5. 刪除「購買須知」「想知道你的電腦能不能跑這款AI模型？」「使用AI硬體相容性檢測工具 ❱ https://...」等固定模板開場白（computer_specs 已是獨立欄位，不必在描述裡重複）
6. 禁止添加原文沒有的資訊
7. 禁止改變語氣風格（不加油、不變好笑、不變浮誇）

原文：
{原文}

請直接輸出修改後的描述，不要加任何解釋。
```

### 為什麼不全部寫死 regex

- hashtag 格式不統一：有的寫 `#AI #NSFW`，有的寫 `#AI视频 #NSFW`，有的藏在句尾沒空格
- 硬體規格的寫法變化大：`8GB顯存`、`8G VRAM`、`建議N卡，顯示8G起` 難以窮舉
- emoji 種類和位置不可預測
- 用 LLM 理解語義來判斷哪些該留哪些該砍，比 regex 準確且好維護

### 風險與對策

| 風險 | 對策 |
|---|---|
| AI 幻覺/亂加內容 | prompt 明文禁止 + 產出後 diff 比對，只允許刪除不允許新增 |
| 誤刪重要資訊 | 每條修改記錄 log，人工可以快速復原 |
| 關閉時完全跳過 | 開關預設 OFF，不跑就不消耗 token，也不影響既有流程 |
